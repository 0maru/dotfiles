# ハンドオフドキュメント仕様

エージェント間の引き継ぎに使用する構造化 Markdown フォーマット。

## テンプレート

```markdown
# Orchestra Handoff: {前フェーズ} → {次フェーズ}

## メタデータ
- ワークフロー: {review | feature | bugfix | refactor}
- フェーズ: {current} / {total}
- イテレーション: {current} / {max}
- タイムスタンプ: {YYYY-MM-DDTHH:MM:SS}
- ステータス: {PASS | REVISE | BLOCK | IN_PROGRESS}

## タスク概要
{元のタスク説明。Phase 1 から変更せずに引き継ぐ}

## 対象
- PR: {PR URL（該当する場合）}
- ブランチ: {ブランチ名}
- ベースブランチ: {main など}

## 検出環境
- 言語: {検出された言語のカンマ区切りリスト}
- 適用スキル: {使用したスキル名のカンマ区切りリスト}

## 変更ファイル
- `{path/to/file}` — {変更の簡潔な説明}

## 発見事項

### CRITICAL
| ファイル | 説明 | ステータス |
|---------|------|----------|
| `file:line` | {説明} | {未修正 / 修正済} |

### HIGH
| ファイル | 説明 | ステータス |
|---------|------|----------|
| `file:line` | {説明} | {未修正 / 修正済} |

### MEDIUM
| ファイル | 説明 | ステータス |
|---------|------|----------|
| `file:line` | {説明} | {未修正 / 修正済} |

### LOW
| ファイル | 説明 | ステータス |
|---------|------|----------|
| `file:line` | {説明} | {未修正 / 修正済} |

## サマリー
- CRITICAL: {N} 件
- HIGH: {N} 件
- MEDIUM: {N} 件
- LOW: {N} 件

## 判定
{PASS | REVISE | BLOCK}

{判定理由の1-2文の説明}

## 次フェーズへの推奨事項
- {具体的なアクション項目}
```

## フィールド説明

### ステータス値

| 値 | 意味 | 次のアクション |
|---|---|---|
| PASS | 全基準を満たす | 次フェーズに進む |
| REVISE | 修正可能な問題あり | 実装エージェントに差し戻し |
| BLOCK | 根本的な問題 / 人間判断が必要 | ユーザーにエスカレーション |
| IN_PROGRESS | 処理中 | 待機 |

### Severity 基準

既存の `security-review` スキルおよび `/code-review` コマンドと同じ基準:

| Severity | 基準 | 例 |
|---|---|---|
| CRITICAL | データ漏洩・認証バイパス・RCE | SQLインジェクション、ハードコードされた秘密鍵 |
| HIGH | 権限昇格・情報漏洩・重大なバグ | IDOR、エラーハンドリング欠如、50行超の関数 |
| MEDIUM | ベストプラクティス違反 | ミューテーションパターン、テスト欠如 |
| LOW | スタイル・改善推奨 | 命名の不一貫、型アノテーション欠如 |

### 揺動検出

evaluator が同じファイル・同じ行に対して同一カテゴリの指摘を2回以上行った場合:
- REVISE → BLOCK に昇格
- 理由に「揺動パターン検出」と記載
- loop-operator はループを停止し、ユーザーにエスカレーション

## ファイル命名規則

```
plans/orchestra-handoff-{workflow}-{YYYYMMDD-HHMMSS}.md
```

例: `plans/orchestra-handoff-review-20260327-143022.md`
