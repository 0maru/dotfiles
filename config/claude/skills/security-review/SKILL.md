# セキュリティレビュー

## 必須プリコミットチェック

コミット前に以下を確認：
- [ ] ハードコードされたシークレット（APIキー、パスワード、トークン）がないか
- [ ] ユーザー入力がすべてバリデーションされているか
- [ ] SQLインジェクション対策（パラメータ化クエリ）があるか
- [ ] XSS 対策（HTML サニタイズ）があるか
- [ ] CSRF 対策が有効か
- [ ] 認証・認可チェックがあるか
- [ ] レート制限が設定されているか
- [ ] エラーメッセージに機密情報が漏れていないか

## OWASP Top 10 チェックリスト

### A01: アクセス制御の不備
- すべてのエンドポイントに認証チェックがあるか
- ロールベースのアクセス制御（RBAC）が実装されているか
- 他ユーザーのリソースにアクセスできないか（IDOR）
- API エンドポイントの認可が正しいか

### A02: 暗号化の失敗
- 機密データが暗号化されているか（保存時・転送時）
- パスワードが適切にハッシュされているか（bcrypt, argon2）
- 弱い暗号化アルゴリズムが使われていないか（MD5, SHA1 for passwords）
- TLS が強制されているか

### A03: インジェクション
- SQL クエリがパラメータ化されているか
- コマンド実行にユーザー入力が含まれていないか
- XSS 対策（出力エンコーディング）があるか
- LDAP, XML, XPATH インジェクション対策があるか

### A04: 安全でない設計
- 脅威モデリングが考慮されているか
- ビジネスロジックにセキュリティ制約があるか
- レート制限が実装されているか
- 入力バリデーションが多層で行われているか

### A05: セキュリティの設定ミス
- デバッグモードが本番で無効か
- デフォルトの認証情報が変更されているか
- 不要な機能・エンドポイントが無効化されているか
- セキュリティヘッダー（CSP, X-Frame-Options 等）が設定されているか

### A06: 脆弱で古いコンポーネント
- 依存関係に既知の脆弱性がないか（`npm audit`, `pip audit` 等）
- 使用していない依存関係が削除されているか
- フレームワーク・ライブラリが最新か

### A07: 認証の不備
- パスワードポリシーが適切か
- 多要素認証がサポートされているか
- セッション管理が適切か（タイムアウト、無効化）
- ブルートフォース対策（アカウントロックアウト、遅延）があるか

### A08: ソフトウェアとデータの整合性の不備
- CI/CD パイプラインが保護されているか
- 依存関係の整合性が検証されているか
- シリアライゼーションが安全か

### A09: ログとモニタリングの不備
- セキュリティイベントがログに記録されているか
- ログに機密情報が含まれていないか
- アラート・モニタリングが設定されているか

### A10: サーバーサイドリクエストフォージェリ (SSRF)
- 外部URLへのリクエストがバリデーションされているか
- 内部ネットワークへのリクエストがブロックされているか
- URL のホワイトリストが実装されているか

## シークレット管理

- **絶対に**シークレットをソースコードにハードコードしない
- 環境変数またはシークレットマネージャーを使用
- 必要なシークレットがアプリ起動時に存在することを検証
- 漏洩した可能性のあるシークレットは即座にローテーション

## セキュリティレスポンスプロトコル

CRITICAL な脆弱性を発見した場合：
1. **即座に停止** — 他のコード変更を中断
2. **報告** — 問題の概要と影響範囲を明確に記述
3. **修正例を提示** — 安全なコードの具体例を示す
4. **検証** — 修正後にセキュリティテストを実行
5. **シークレット漏洩の場合** — 即座にローテーションを推奨

## Severity 基準

| Severity | 説明 | 対応 |
|----------|------|------|
| CRITICAL | データ漏洩・認証バイパス・リモートコード実行のリスク | 即座に修正。リリースブロック |
| HIGH | 権限昇格・情報漏洩のリスク | リリース前に修正 |
| MEDIUM | 設定ミス・ベストプラクティス違反 | 次のスプリントで修正 |
| LOW | 改善推奨 | バックログに追加 |
