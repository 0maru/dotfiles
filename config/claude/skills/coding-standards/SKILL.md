# コーディング標準

## ファイル構成

- **多くの小さなファイル**: 200-400行が目安、800行を超えない
- **機能ごとに分割**: 型別（models/, controllers/）より機能別（users/, orders/）
- **1ファイル1責務**: 1つのファイルが1つの明確な責務を持つ

## 関数サイズ

- **50行以下**: 1つの関数が50行を超えたら分割を検討
- **1つの関数1つの目的**: 関数名で目的が説明できること
- **ネスト深度4以下**: 4段階を超えるネストは早期リターンやヘルパー関数で解消

## 不変性

- 既存のオブジェクトを変更せず、新しいオブジェクトを作成する
- 更新操作はコピーを返す（元のデータは変更しない）
- 言語の不変性機能を活用（`const`, `readonly`, `frozen`, `final` 等）

## エラーハンドリング

- **全レベルで明示的にハンドリング**: エラーを握りつぶさない
- **UI コード**: ユーザーフレンドリーなメッセージ
- **サーバーコード**: 詳細なサーバーサイドログ
- **コンテキスト情報**: エラーメッセージに「何が」「なぜ」失敗したかの情報を含める
- **適切なエラー型**: 言語のエラー/例外機構を適切に活用

## 入力バリデーション

- **システム境界で検証**: 外部データ（API入力、フォーム、ファイル読み込み）は受け取り時にバリデーション
- **早期失敗**: 不正なデータは処理の早い段階で拒否
- **明確なエラーメッセージ**: どのフィールドがなぜ不正かを明示

## 命名

- **明確で説明的**: 略語を避け、目的がわかる名前
- **一貫性**: プロジェクト内で命名規則を統一
- **スコープに応じた長さ**: ループ変数は短く（`i`, `j`）、グローバルは長く説明的に
- **ブール値**: `is`, `has`, `should`, `can` プレフィックス
- **コレクション**: 複数形（`users`, `items`）
- **ハードコード値禁止**: マジックナンバーは定数に抽出

## 品質チェックリスト

- [ ] 関数が50行以下か
- [ ] ファイルが800行以下か
- [ ] ネスト深度が4段階以下か
- [ ] ハードコード値がないか（定数化されているか）
- [ ] 不変パターンが一貫して適用されているか
- [ ] 命名が明確で読みやすいか
- [ ] エラーが明示的にハンドリングされているか
- [ ] 外部入力がシステム境界でバリデーションされているか
