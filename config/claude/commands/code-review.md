# /code-review

$ARGUMENTS の変更をレビューする。引数がない場合は `git diff HEAD` の変更をレビューする。

## 手順

1. **変更ファイルの取得**: `git diff --name-only HEAD` を実行
2. **各ファイルの差分を分析**: `git diff HEAD` で具体的な変更内容を確認
3. **チェックカテゴリに従ってレビュー**
4. **構造化されたレポートを出力**

## チェックカテゴリ

### セキュリティ（CRITICAL）
- ハードコードされた認証情報・APIキー・トークン
- SQLインジェクション脆弱性
- XSS 脆弱性
- 入力バリデーションの欠如
- 脆弱な依存関係
- パストラバーサルリスク
- 認証・認可の欠陥

### コード品質（HIGH）
- 50行を超える関数
- 800行を超えるファイル
- 4段階を超えるネスト
- エラーハンドリングの欠如
- console.log / print 文の残留
- TODO/FIXME コメント

### ベストプラクティス（MEDIUM）
- ミューテーションパターン（不変パターンを推奨）
- 不要な複雑さ
- 新コードに対するテストの欠如
- パフォーマンスの懸念

### スタイル（LOW）
- 一貫性のない命名
- 型アノテーションの欠如
- フォーマットの問題

## レポートフォーマット

問題ごとに以下の形式で報告：

```
**[SEVERITY]** file.ts:123
Issue: [説明]
Fix: [修正方法]
```

## 判定

- **CRITICAL または HIGH の問題あり**: コミット前に修正必須
- **MEDIUM の問題のみ**: マージ前に修正推奨
- **LOW の問題のみ**: 任意の改善

## 注意事項

- 信頼度80%以上の問題のみ報告する
- 同種の問題は統合して報告
- セキュリティ脆弱性は絶対に見逃さない
