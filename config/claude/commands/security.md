# /security

$ARGUMENTS のコードをセキュリティ観点でレビューする。引数がない場合は `git diff HEAD` の変更を対象とする。

## 手順

1. **対象コードの特定**: 変更ファイルまたは指定されたファイル/ディレクトリを読み込む
2. **OWASP Top 10 に基づくチェック**を実施
3. **シークレットスキャン**を実行
4. **構造化されたレポートを出力**

## チェック項目

### インジェクション
- パラメータ化クエリが使われているか
- ユーザー入力がサニタイズされているか
- コマンド実行にユーザー入力が含まれていないか

### 認証・認可
- パスワードが適切にハッシュされているか（bcrypt/argon2）
- セッション管理が適切か
- すべてのエンドポイントに認証チェックがあるか
- ロールベースのアクセス制御が正しいか

### 機密データ
- 暗号化が適切に設定されているか
- PII の取り扱いが適切か
- ログに機密情報が含まれていないか

### XSS
- 出力エンコーディングが行われているか
- CSP ヘッダーが設定されているか
- ユーザー入力がHTML挿入に使われていないか

### 設定
- CORS が適切に設定されているか
- レート制限が設定されているか
- CSRF 対策が有効か
- Cookie に secure / httpOnly / sameSite フラグがあるか

### シークレット
- ソースコードにAPIキー・パスワード・トークンが含まれていないか
- 環境変数またはシークレットマネージャーが使われているか

## レポートフォーマット

### 1. Critical Issues（即座に対応が必要）
```
**[CRITICAL]** file.ts:123
Vulnerability: [脆弱性の説明]
Impact: [影響範囲]
Remediation: [修正方法]
```

### 2. High Priority（リリース前に修正）
```
**[HIGH]** file.ts:456
Issue: [問題の説明]
Fix: [修正方法]
```

### 3. Recommendations（改善提案）
```
**[MEDIUM]** file.ts:789
Suggestion: [提案内容]
```

## 判定基準

- **CRITICAL が1つでもある場合**: リリースブロック。即座に修正が必要
- **HIGH がある場合**: リリース前に修正が必要
- **MEDIUM のみ**: 次のスプリントで対応
