# Commit Push PR

## 概要

差分の棚卸しから PR URL の返却までを一気通貫で進める。変更を無差別に 1 つへまとめず、目的ごとに分けてコミットし、日本語のコミットメッセージと PR 文面で仕上げる。

## 手順

### 1. 現状を把握する

- 現在ブランチ、`git status`、`git diff HEAD`、直近コミット、`origin/main..HEAD` の差分コミットを確認する。
- 可能なら読み取り系コマンドは並列に実行して、変更の全体像を先に掴む。
- 変更内容を理解する前に `git add .` や 1 回の雑なコミットへ進まない。

### 2. コミット対象を仕分ける

- 変更を目的、ツール、機能単位でグループ化する。
- 例: `aerospace` 設定、`codex` 設定、`zsh` 設定は別コミット候補として扱う。
- シークレット、認証情報、`.env`、credentials 類はコミットしない。
- ユーザーが意図していない無関係変更は含めない。submodule ポインタ変更なども同様に扱う。
- すべてが同じ目的の変更なら 1 コミットにまとめてよい。

### 3. 必要ならブランチを切る

- 現在ブランチが `main` の場合は、変更内容を説明する名前で新しいブランチを作成する。
- 既存ブランチにいる場合は、そのブランチを使い続ける。明示指示がない限り勝手に作り直さない。
- Git は非対話コマンドで扱い、`git switch -c <branch>` を優先する。

### 4. 論理単位でコミットする

- 各グループごとに必要なファイルだけを stage して個別にコミットする。
- コミットメッセージは日本語で書く。
- タイトルは簡潔に 50 文字以内を目安にする。
- 変更意図が diff だけで読みにくい場合は本文を付ける。
- 本文では「何を」「なぜ」変えたか、将来の読者に役立つ文脈を補う。
- 空コミットは作らない。コミットすべき変更がない場合は次の手順へ進む。

### 5. push する

- 現在ブランチを `origin` へ push する。
- 追跡設定がない場合は `git push -u origin HEAD` を使う。
- 新規コミットがない場合でも、未 push のコミットがあるなら push を行う。

### 6. PR を作成する

- `gh pr create` を使って PR を作成する。
- タイトルは日本語で 70 文字以内、ブランチ上の変更全体を要約する。
- 本文は次の形式を使う。

```md
## 概要
- <ブランチ上の変更をまとめた 1〜3 行の箇条書き>

## テスト計画
- [ ] <確認項目>
```

- 既存 PR がある場合は重複作成を避け、既存の PR URL を返す。

## 出力

- 実行後は以下を短くまとめて返す。
  1. 作成または利用したブランチ名
  2. 作成したコミット一覧
  3. push の結果
  4. PR URL
- 除外した変更や、判断保留にした差分があれば明示する。

## 失敗時の扱い

- `gh` が使えない、認証切れ、push 権限がないなどで完了できない場合は、どこまで終わったかと失敗原因を明示する。
- 破壊的な git 操作で帳尻を合わせない。`reset --hard` や無断の `checkout --` は使わない。
